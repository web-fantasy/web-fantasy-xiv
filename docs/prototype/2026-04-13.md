# XIV Stage Playground

这是一个模拟《最终幻想14（Final Fantasy XIV）》游戏中 BOSS 战斗的模拟器，旨在打造非常类似的战斗体验，提供一个平台让玩家可以练习和测试他们的技能和策略。甚至完全自定义 BOSS 的技能和行为。

我们首先要对FF14的各种机制设计进行分析和理解。

## 组件的解耦

我们在编程时，要尽量将不同的功能模块进行解耦，这样可以提高代码的可维护性和可扩展性。

比如：场景 map、实体 entity、技能 skill、buff/debuff、伤害计算 damage calculation 等等。

## 事件系统

我们需要提前设计好事件系统，来处理各种游戏事件，比如技能释放、伤害计算、状态变化等等。这样可以让我们的代码更加清晰和易于维护。

另外，也可以预留一些（比如基于 EventEmitter）事件接口，比如：当某个技能造成伤害时触发一个事件，上下文携带攻击者、受击者、技能、伤害数值等信息。这样我们在后期就可以在其他地方监听这个事件，来实现一些额外的功能，比如统计伤害、触发某些特殊效果等等。

## 伤害计算

### 基础攻击力

我们使用简化的伤害计算系统。

所有实体均有一个基础攻击力数值，所有的技能伤害均指代百分比。

例如：玩家基础攻击力1,000，使用了威力200的技能，那么这个技能造成的伤害就是1,000 \* 200% = 2,000。

我们也可以把敌方实体的默认基础攻击力设置为1，这样在设计BOSS时更方便，因为BOSS的技能威力直接等于它造成的伤害数值，设计起来更直观。

### 增伤

增伤的计算方式总是相加，而不是相乘。
实际伤害 = 原始伤害 x (1 + 增伤A + 增伤B + ...)
例如：玩家同时受到增伤率 50% 的效果 + 增伤率 30% 的效果，那么玩家受到的实际伤害 = 原始伤害 x (1 + 50% + 30%) = 原始伤害 x 1.8。

### 减伤

我们也使用简化的减伤系统。实体自身没有防御力，即威力 1,000 的技能命中默认就会造成 1,000 的伤害。

FF14减伤计算核心为相乘关系，即多种减伤技能叠加时，后续减伤是在前一个减伤后的剩余伤害基础上再次减免。
实际伤害 = 原始伤害 x (1 - 减伤A) x (1 - 减伤B) ...
也就是说，玩家同时使用减伤率 80% 的技能 + 减伤率 20% 的技能，并不会获得“完全防御”，而是依然会受到原始伤害的16%（= 原始伤害 x (1 - 80%) x (1 - 20%) = 原始伤害 x 0.2 x 0.8 = 原始伤害 x 0.16）。

## 技能系统

FF14的技能有且只有以下几种类型：战技、魔法、能力技。

其中，战技与魔法会触发一个公共冷却时间，被称为 Global Cooldown（GCD），默认为2.5s。能力技则不受 GCD 的限制，可以在 GCD 期间使用。

### 战技

无需咏唱时间，使用后会触发 GCD。

只需在使用的瞬间判定是否满足使用条件（比如距离、状态等），如果满足则立即生效。

### 魔法

与战技的区别在于，魔法需要咏唱时间。

当魔法开始咏唱时，会立即触发 GCD，但魔法的效果要等到咏唱完成后才会生效。

魔法类技能需要在咏唱时和咏唱完成时进行两次判定，分别检查使用条件和效果生效条件。否则魔法会释放失败（不扣除 MP，但也不造成任何效果）。

咏唱：咏唱期间，玩家不能使用其他技能（包括能力技），任何移动或调整面向将会打断咏唱。

打断咏唱：玩家可以使用ESC、移动、调整面向等方式来打断咏唱。咏唱被打断时，技能释放失败，不扣除 MP，但也不造成任何效果，立即重置 GCD。

默认情况下，魔法的咏唱时间等于技能的 GCD 时间，即魔法咏唱结束后GCD技能会正好冷却完成。但也存在一些特殊情况：

1. 咏唱时间 > GCD：假设咏唱时间为3s，GCD为2.5s，在第2.6s时，GCD已经冷却完成，但魔法仍在咏唱中，因此玩家依然无法使用其他技能，直到第3s时咏唱完成，魔法生效后玩家才能使用其他技能。除非玩家选择打断咏唱，此时魔法释放失败，但GCD已经冷却完成，玩家可以立即使用其他技能。
2. 咏唱时间 < GCD：假设咏唱时间为2s，GCD为2.5s，在第2s时，魔法咏唱完成，魔法生效后玩家仍然需要等待0.5s才能使用其他GCD技能。除非玩家选择在2s之内（魔法释放之前）打断咏唱，此时魔法释放失败，GCD立即重置，玩家可以立即使用其他技能。

### 能力技

能力技有各自独立的CD时间，能力技不受 GCD 的限制，可以在 GCD 期间使用。

能力技不受沉默影响（沉默只会禁止战技和魔法的使用），但仍然需要满足其他使用条件（比如距离、状态等）。

能力技的使用条件和效果生效条件通常在使用的瞬间进行判定，如果满足则立即生效。

---

但是也存在以下特殊的技能，它们不存在于玩家或实体的技能列表里，更多是功能用途：

### 自动攻击

自动攻击是指玩家或实体在满足一定条件下会自动进行的单体攻击行为，通常不需要主动触发。

所有实体均拥有一个自动攻击间隔，默认为3s。在进入战斗状态时立即触发第一次自动攻击，之后每隔自动攻击间隔触发一次，直到脱离战斗状态。

所有实体均拥有一个自动攻击距离，默认为5m。每次自动攻击判定时，总是检查玩家当前锁定的目标是否位于正前方180度且距离在自动攻击距离内，如果满足条件，则自动攻击生效；如果不满足条件，则自动攻击失败，但不会重置自动攻击的计时器，下一次自动攻击仍然会在自动攻击间隔后触发。

自动攻击的判定类似于自动触发的战技：每次自动攻击触发时，都会进行一次判定，检查当前是否满足自动攻击的使用条件（比如距离、状态等）。如果满足条件，则自动攻击生效；如果不满足条件，则自动攻击失败，但不会重置自动攻击的计时器，下一次自动攻击仍然会在自动攻击间隔后触发。

以下情况会导致自动攻击失败：

1. 目标距离过远：如果目标距离超过自动攻击的最大范围，则自动攻击无法命中目标，自动攻击失败。
2. 目标状态不适合：如果目标处于某些特殊状态（比如隐身、无敌等），则自动攻击无法对其造成伤害，自动攻击失败。
3. 正在咏唱：如果玩家正在咏唱魔法，则自动攻击无法触发，因为咏唱期间无法使用其他技能，自动攻击失败。

但是自动攻击可以和战技与能力技同时触发：例如玩家在第3s时释放了一个战技，此时只要满足条件，自动攻击和战技会同时生效，并各自造成伤害。

敌方实体（甚至BOSS）也默认拥有自动攻击能力，除非BOSS正在咏唱魔法，否则它的自动攻击会正常触发。

### 特殊攻击

特殊攻击又被称为“系统杀”，一般用于场地伤害或者所谓的“狂暴技能”，它们在任何时候生效，不进行使用条件的判定，不吃任何减伤效果，直接对目标造成伤害。

举例：
玩家走出了战斗场地（或者掉出了地图，Z轴值太小了），这时玩家会持续受到威力999,999,999的特殊伤害，这在一般情况下会立即杀死玩家，以便重置战斗；
另一个广泛的用途是，BOSS战进入尾声，玩家依然没有击败BOSS，这时BOSS会咏唱所谓的“狂暴技能”，它的咏唱时间会很久，甚至长达数十秒，咏唱完毕后，系统会同时为所有玩家叠加一个威力999,999,999的特殊伤害，强制击杀玩家，防止玩家用特殊手段规避掉BOSS的魔法伤害，最终导致战斗无法结束。

## 敌方实体AI

敌方实体（尤其是BOSS）的AI设计是整个模拟器的核心部分之一。我们需要为敌方实体设计一套智能的行为模式，使其能够根据战斗情况做出合理的决策。

实体AI一般由一个状态机驱动，状态机包含多个状态（比如待机、游走、战斗、脱战等），每个状态都有自己的行为逻辑和转换条件。

所幸，FF14的BOSS战斗大多数时候遵循固定时间轴设计，即BOSS会在特定的时间点触发特定的技能或行为，这使得我们可以通过预设时间轴来实现BOSS的AI行为，而不需要过于复杂的动态决策算法。

### 待机/游走

实体没有进入战斗状态时，处于待机或游走状态。此时实体会在场地内随机移动，或者保持原地不动，等待玩家接近。

而BOSS战场景中我们可以忽略游走策略，在BOSS被初始化后固定在场地中央不动，也不转动面向，直到玩家进入战斗状态。

### 引战

敌方实体一般在以下情况下进入战斗状态：

1. 玩家进入实体的仇恨范围：当玩家进入敌方实体的仇恨范围时，敌方实体会立即进入战斗状态，开始攻击玩家。
2. 玩家对敌方实体造成伤害：当玩家对敌方实体造成伤害时，敌方实体会立即进入战斗状态，开始攻击玩家。

仇恨范围：一般来说是一个扇形区域，默认是以实体为中心，半径=普通攻击的最大攻击范围，角度=120度。一旦玩家走进这个区域，敌方实体就会进入战斗状态。

### 战斗中

敌方实体在战斗状态下会根据预设的时间轴触发特定的技能或行为。

索敌：敌方在常规攻击阶段总是会尝试面向它要攻击的玩家，如果玩家走出它的自动攻击范围，敌方总是会尝试找到一条最短路径，使玩家位于自己自动攻击距离-0.1m的范围内，并且面向玩家。

除非它正在咏唱魔法，BOSS一旦开始咏唱魔法，期间总是不会移动或调整面向，就和玩家一样，只是它们永远不会尝试打断咏唱，即使玩家已经走出技能范围（否则玩家会完全无法规避伤害）。

BOSS的攻击可能会失效，比如玩家在BOSS释放技能的瞬间已经走出了技能范围，或者玩家在BOSS咏唱魔法的过程中打断了咏唱，这些都会导致BOSS的攻击失效，但这并不影响BOSS后续的攻击行为，BOSS仍然会按照时间轴继续进行攻击。

以下与程序设计无关：

- 我们需要在设计BOSS时间轴时规避一些糟糕的设计，比如让BOSS咏唱单体攻击魔法，但是施法距离设计的很短，这会导致BOSS一直尝试拉进与玩家的距离，或者咏唱结束时玩家跑开了，这会让魔法失效；我们要么就让BOSS的单体攻击都是战技，要么就让BOSS的单体攻击魔法的施法距离足够大，或者设计成范围攻击，这样就不会出现BOSS一直尝试拉近距离或者魔法失效的情况。范围攻击总是被描述为“在脚下释放”，所以BOSS不会在释放之前首先尝试索敌。
- 对于那些不以BOSS脚下为中心的范围攻击，我们可能需要一些独特的设计，例如召唤一个隐藏的不可被选中的临时实体，让它在特定位置在脚下释放范围攻击，这样就不会出现BOSS一直尝试面向释放位置或者索敌的情况。
- 尽量不要给BOSS设计战技，因为大多数时间我们不希望检查BOSS的技能CD，否则无法设计出多次技能连续判定的演出，所以我们可以给BOSS设计许多冷却时间为0的能力技来代替战技，这样就不需要检查CD了。
- 有时候我们想要设计出这种效果：BOSS咏唱后释放多段伤害，这个时候的技能安排取决于我们想不想让BOSS在此期间可以移动，或者范围伤害受不受BOSS的位置、面向影响，大多数时候可以设计成：BOSS咏唱10s魔法，然后立即连续咏唱多次0.5s的魔法，这样BOSS在此期间就不会移动；但有时候设计上是BOSS在脚下扔了一个炸弹，炸弹会爆炸好几次，这可能需要召唤一批无AI临时实体来实现。
- 总之：尽量不要破坏技能的基本设计原则（比如技能需要满足使用条件才能生效，技能需要在使用的瞬间进行判定，技能需要在生效的瞬间进行判定等等），否则就会导致BOSS的行为模式与技能设计需要大量的额外编程与耦合设计。

关于时间轴的设计，是我们后面要详细讨论的一个重要话题，它需要足够灵活，但又可服用性强，能够快速直观地描述BOSS的行为模式。

以最简单的小怪为例，它的时间轴可能是：每隔5s使用一次战技A，每隔10s使用一次战技B。
那么它的战斗行为：尝试面向玩家，尝试拉近与玩家的距离，使玩家位于自动攻击范围内，然后每隔5s使用一次战技A，每隔10s使用一次战技B。

## 实体朝向/面向

所有实体都有一个朝向属性，表示实体当前面向的方向。

正前方左右45度被称为“正面”/“前面”forward，正后方左右45度被称为“背面”back，正左方左右45度被称为“左侧”left，正右方左右45度被称为“右侧”right。

释放技能时，如果技能需要一个目标，那么实体一定会面向那个目标。在玩家释放技能的瞬间，我们需要自动调整玩家的面向，使其朝向目标。对于敌人实体来说也是一样的，如果技能需要一个目标，那么它们在释放技能的瞬间也会自动调整面向。
如果技能的目标不是另一个实体，而是一个位置，那么实体会面向那个位置。
只有一种情况下实体在释放技能时不会调整面向：技能没有目标/目标在自己脚下/目标是自己，例如在脚下释放圆形AOE/为自己回血。
另外，自动攻击不会自动调整面向，如果自动攻击范围内没有可攻击的目标则该次自动攻击无效。

特殊情况：为了演出效果，BOSS的某些技能会出现固定朝向的技能，例如BOSS瞬移回到场中，固定面向南方，释放左右刀，此时需要冻结BOSS的索敌AI，以免BOSS在左右刀的间隙里自动面向玩家，导致左右刀的效果歪斜。写时间轴文件时要注意这一点。

## 技能范围

技能的效果范围分为：单体技能、范围技能。

单体技能比较简单，因为技能已经自带了能否释放的检测，所以单体技能可以被认为是一定会命中的。

范围技能的种类十分多样，可以说是FF14战斗设计最具创造力的部分了，我们可以把范围技能拆解为以下基本形状：

- 圆形：以某个点为中心，半径为R的圆形区域。
- 扇形：以某个点为中心，半径为R，夹角为A的扇形区域。（圆形可以看作是夹角为360度的扇形？）
- 环形：以某个点为中心，内半径为R1，外半径为R2的环形区域。（圆形可以看作是内半径为0的环形？）
- 矩形：以某个点为中心，长度为L，宽度为W，朝向为D的矩形区域。

由此基础上可以衍生出FF14几乎所有的组合型范围技能设计，例如：

- 左右刀：BOSS向左侧释放一个超大范围的180度扇形技能，在短暂的间隔后再向右侧释放一个超大范围的180度扇形技能，这样就形成了一个稍有时间间隔的左右对称的范围攻击。玩家首先呆在BOSS右侧，然后在第一个扇形技能的间隔里跑到BOSS左侧，来规避掉两个扇形技能的伤害。
- 钢铁月环：BOSS首先释放一个以自身为圆心的圆形AOE，在短暂的间隔后再释放一个以自身为圆心内半径等于第一个圆形AOE半径，外半径更大的环形AOE，这样就形成了一个稍有时间间隔的内外同心圆的范围攻击。玩家首先需要远离BOSS，然后在第一个圆形AOE的间隔里跑到BOSS附近，来规避掉两个圆形AOE的伤害。
- 十字切：以BOSS为中心的十字形范围技能，实现方案比较多样，例如可以在场地边缘召唤两只临时隐形实体释放两个直线AOE来实现，也可以让BOSS向四个方向分别释放四个矩形AOE来实现，取决于我们是否希望站在BOSS脚下的玩家也受到伤害，以及我们是否希望十字切的范围受BOSS的面向影响。
- 九字切：场地中间出现多条横纵交替的矩形AOE，形成类似井字棋的范围攻击，然后另一批矩形AOE在第一个矩形AOE的间隔里出现，正好覆盖第一次攻击的安全区。玩家必须首先站在第一个矩形AOE的安全区里，然后在第一个矩形AOE的间隔里跑到第二个矩形AOE的安全区里，来规避掉两批矩形AOE的伤害。

## 技能预兆/生效提示

技能预兆是指在技能生效之前，给玩家一个视觉上的提示，让玩家知道这个技能即将生效，并且这个技能的效果范围在哪里。

只有敌方实体释放的范围技能才会有技能预兆，玩家释放的范围技能不会有技能预兆（也可以设计为所有范围技能都有预兆，但默认关闭，以确保封装上的一致性）

技能预兆是技能效果范围在其下方地图模型上的投影，通常以缓慢闪烁的半透明橙色区域+透明度稍低的边框线的形式呈现，其持续时间是可配置的。

技能的预兆提前多久显示/是否显示应被设计为可配置的，因为我们有时候更希望玩家通过观察BOSS的动作来判断技能范围，如果提前给出预兆就像开卷考一样无趣。为了防止忘记配置，默认可以是咏唱开始时即显示预兆。

生效提示：范围技能在生效的瞬间会变为红色，默认持续0.5s后消失。

技能的红色生效提示最好默认是开启的，这样方便玩家判断自己最终被什么技能击中了，来分析自己为什么死了。当然，这个可以被配置为不显示，因为我们可能使用动画或贴图来替代纯粹的红色提示。

## 判定

我们使用简化的判定，所有实体只有其脚底中心点为技能判定点，而不考虑其体积。

比如计算技能的攻击距离，只计算玩家中心点与目标中心点的距离。

所有的范围技能判定只考虑其投影面积，也就是说没有Z轴，即使玩家跳跃起来了，只要其在范围技能的投影范围内，就会被判定为命中。

碰撞判定也采用简单判定，这虽然会导致玩家靠墙时身体的一部分建模穿入墙内，但是无伤大雅。

实体之间没有碰撞，可以互相穿模。

## 玩家移动

玩家可以通过键盘+鼠标输入来控制移动

- 默认使用WASD键来控制基于当前相机朝向的前后左右移动
- 按下鼠标右键拖动来调整相机的朝向
- 面向调整有两种模式切换：
  - 经典模式：玩家的面向总是与相机朝向一致，即使侧向移动或后退，面向也始终朝向相机的正前方
  - 独立模式：玩家会自动朝向行走方向，例如相机对着正北，按下S时玩家会朝向正南，按下A时玩家会朝向正西，按下D时玩家会朝向正东，按下W时玩家会朝向正北
  - 但是释放技能的瞬间我们总是会尝试自动调整一次玩家的面向，使其朝向技能目标或技能目标位置，这样就不需要玩家在释放技能前刻意调整面向了。
  - 关于背对类技能：有些BOSS会释放所谓的“不可直视”的技能，例如美杜莎的石化凝视，判定起来也很简单，只需将玩家位置与BOSS位置连线，然后计算此连线与玩家当前面向的夹角，大于90度可以被认为是背对。
- 玩家移动的速度是可配置的，默认值可以设置为FF14中玩家的平均移动速度，例如5m/s。加速类技能总是百分比提速，且不能叠加，只取加成最多的那一个，例如玩家同时受到加速率50%和加速率30%的效果，那么玩家的实际移动速度 = 基础移动速度 x (1 + 50%) = 基础移动速度 x 1.5。加速和减速效果之间总是加算，例如玩家给自己30%加速，被BOSS减速50%，那么玩家的实际移动速度 = 基础移动速度 x (1 + 30% - 50%) = 基础移动速度 x 0.8。
- 默认使用空格跳跃，跳跃的高度是可配置的，但是跳跃轨迹和距离需要在跃起瞬间进行一个计算，玩家在半空中只能调整面向，而无法调整轨迹，也就是说在跃起的瞬间其轨迹就被确定了，而不能跳出一个诡异的S形。

## BOSS时间轴

BOSS时间轴是用来描述BOSS在战斗中行为模式的工具，它可以让我们以一种直观的方式来设计和调整BOSS的行为。

这部分需要我们单独去设计，可能是一个json/yaml，也可以是一个我们独创的脚本语言

它需要可以在开头描述一些全局的设置，例如BOSS的初始HP、BOSS的自动攻击间隔、BOSS的自动攻击距离、BOSS的仇恨范围等等，这些全局设置会直接影响BOSS的行为模式，例如自动攻击间隔和自动攻击距离会影响BOSS的索敌行为，仇恨范围会影响BOSS何时进入战斗状态等等。

也可以定义一些技能模板，BOSS可以在一场战斗中释放多次同一个技能，这时就可以在时间轴开头定义一个技能模板，然后在时间轴里直接引用这个技能模板来释放技能，这样就不需要每次都写一大段技能配置了。
